009: 摘要
010: 本报告围绕智慧蘑菇农场项目的单元测试工作展开，覆盖 Web 后端、Web 前端和鸿蒙终端三个层面。报告从测试目标、测试环境、测试策略、测试对象、代表性测试用例、执行结果、覆盖率分析和改进建议等方面进行系统整理。
011: 当前测试记录显示，Web 后端共 13 个测试类、116 个测试方法，Web 前端共 3 个测试文件、46 个用例，均已执行通过；鸿蒙端已完成 10 个 Hypium 测试文件、约 145 个用例的编写与注册，最终通过截图需在 DevEco Studio 或 hvigorw 环境中补充。报告使用 Mock 隔离数据库、云服务和外部 AI 服务等依赖，重点验证核心业务逻辑和边界处理，形成可重复的质量基线。
012: 关键词：单元测试；JUnit 5；Vitest；Hypium；JaCoCo；Mock；智慧蘑菇
013: 1. 引言
014: 1.1 编写背景
015: 智慧蘑菇农场项目是东北大学软件学院实训课程中的综合实践系统，包含 Web 管理后台、鸿蒙移动终端和嵌入式采集端等组成部分。系统围绕蘑菇种植场景，实现传感器数据展示、设备管理、命令下发、AI 种植问答、IoT 云平台对接等功能。
016: 本次实训要求针对项目核心代码编写单元测试，主要使用 JUnit 5 与 Vitest 完成后端与前端测试，并通过断言验证正常、异常和边界三类场景。为体现项目多端协同能力，本报告同时纳入鸿蒙端 Hypium 单元测试的设计与编写情况。
017: 1.2 文档目的
018: 说明单元测试的设计思路、执行过程和覆盖范围。
019: 为实训答辩、成绩评定和后续维护提供可追溯依据。
020: 帮助团队成员理解核心模块的测试边界、Mock 方式和质量改进方向。
021: 1.3 读者对象
022: 本文档面向指导教师、项目开发成员、测试人员以及后续维护人员。
023: 2. 测试目的与依据
024: 2.1 测试目的
025: 验证核心功能正确性：覆盖设备 CRUD、传感器数据查询、命令记录、AI 问答、健康检查等核心接口与业务逻辑。
026: 掌握单元测试工具链：后端使用 JUnit 5、Mockito、Spring Test 和 JaCoCo；前端使用 Vitest、jsdom 和 coverage-v8；鸿蒙端使用 Hypium。
027: 建立质量基线：通过自动化测试与覆盖率报告记录当前质量状态，为后续迭代提供回归依据。
028: 保证多端逻辑一致：通过前后端镜像用例验证 AI 回答清洗等共享规则。

031: 3. 被测系统概述
032: 3.1 系统架构
033: 智慧蘑菇系统采用“前后端分离 + 移动端 + 嵌入式端”的分层架构。本次单元测试重点覆盖 Web 前端、Web 后端和鸿蒙端的可独立验证逻辑，嵌入式固件和外部云平台交互不作为本次单元测试主体。
034: Web 前端：Vue 3 + Element Plus + ECharts，通过 /api/v1 访问后端。
035: Web 后端：Spring Boot 3.2 + MyBatis-Plus + SQLite，提供 REST API 与 WebSocket。
036: 鸿蒙终端：ArkTS Stage 模型，通过华为云 IoTDA 应用侧 API 同步设备状态。
037: 嵌入式端（WS63）：C 语言固件上报温湿度等数据，本报告仅作为系统背景说明。
039: 图1  智慧蘑菇系统三端架构与单元测试边界
040: 3.2 测试对象在系统中的位置
041: 单元测试不启动完整集成环境，不依赖真实数据库、华为云 IoTDA 或 Dify 服务，而是通过 Mock 与纯函数断言隔离外部依赖，聚焦可重复验证的业务逻辑。
042: HTTP 接口的参数校验、limit 钳制和统一响应格式。
043: Service 层 CRUD、级联删除、默认值填充和缓存优先策略。
044: 工具类字符串处理、Token 缓存、图表数值计算。
045: 前端 API 封装、路由配置与 AI 回答清洗。
046: 鸿蒙 ViewModel 状态管理和异步状态包装行为。

055: 5. 测试策略与方法
056: 5.1 总体策略
057: 本项目采用分层单元测试策略。后端按照 Util/DTO/Holder、Service、Controller 分层推进；前端重点覆盖 utils、api 与 router；鸿蒙端覆盖 ViewModel、工具类和模型常量。集成测试、端到端测试和真实云服务链路测试不在本次实训单元测试范围内。

063: 5.3 Web 前端测试方法
064: 使用 Vitest 作为测试运行器，jsdom 提供浏览器 DOM 环境。
065: 对 axios 使用 vi.mock 进行隔离，验证 API 函数调用的 URL、Method 与参数。
066: 本次不展开 Vue 页面组件测试，Dashboard 等页面对 ECharts / Element Plus 依赖较多，更适合后续使用组件测试或 E2E 测试覆盖。
067: 5.4 鸿蒙端测试方法
068: 使用 Hypium 框架，测试文件位于 entry/src/test/*.test.ets。
069: 通过 List.test.ets 聚合注册全部测试套件。
070: ViewModel 测试以实例化和公开状态断言为主；IoT 不可用时 loadData() 走 Mock 降级路径。
071: UI 页面与 Ability 生命周期测试适合放入 ohosTest 或设备测试，本次未展开。

096: 7.2.1 stripThinking 与后端规则对齐
097: 前端 aiAnswer.test.js 与后端 AiAnswerSanitizerTest 保持一致。
098: 测试目标：避免重复清洗或漏清洗。
099: 7.2.2 API 封装完整性
100: api.test.js 覆盖 20 个导出函数，验证请求方法、路径、参数与响应拦截器行为。例如：
101: getLatest() -> GET /latest。
102: aiChat(question) -> POST /ai/chat，body 为 { question }。
103: 响应拦截器返回 res.data，而不是完整 axios response。
104: 7.3 鸿蒙端代表性用例
105: 7.3.1 DeviceViewModel 在线状态同步
106: 传感器类设备：loadData() 后根据 IotService.isDeviceOnline() 更新 status 与 lastUpdate。
107: 控制器设备（type=控制器）：跳过 IoT 同步，保持 Mock 状态，用例验证 status 与 lastUpdate 不被误改。
108: 7.3.2 GrowthViewModel 批次管理
109: addBatch(GrowthBatch) 使用完整对象参数。
110: archiveBatch(id) 将 status 置为“已采收”，并触发 updateCounts()。
111: loadData() 后 harvestedBatches 由 Mock 统计值重算为当前数据的实际统计结果。
112: 7.3.3 IotTokenManager
113: setToken(token, expiresInSeconds) 使用两参数签名。
114: getToken() 在过期前 1 分钟返回 null，保证后续逻辑可以提前刷新 Token。

136: 10. 覆盖率分析
137: 10.1 Web 后端 JaCoCo 包级覆盖率
139: 图6  Web 后端 JaCoCo 包级覆盖率
140: 当前核心业务包覆盖率较高：Controller 包 96%，Service 包 92%，Util 包 100%。
141: Config、Consumer、WebSocket 等基础设施或集成链路未作为单元测试重点，因此会拉低全项目总覆盖率。
142: 答辩时应强调本次实训目标为核心业务单元测试，而不是对全部源码文件进行同等覆盖。
144: 10.2 Web 前端 Vitest 覆盖率
145: * router funcs 为懒加载 import() 回调，单元测试未触发真实导航；该项不影响对路由配置正确性的验证。
146: 10.3 鸿蒙端覆盖说明
147: Hypium 当前未在本环境生成统一覆盖率 HTML，因此采用“测试文件映射 + 模块计数”的方式描述覆盖情况。
148: ViewModel 层：Dashboard/Alert/Device/Growth 与 BaseViewModel 均有测试文件覆盖。
149: 工具层：ChartUtils、IotTokenManager、ThemeManager 已建立独立测试文件。
150: 常量与模型层：SensorType、Recipe 默认数据等已有较完整测试。

158: 12. 结论
159: 本次智慧蘑菇项目单元测试实训完成了 Web 后端、Web 前端和鸿蒙端三个层面的测试设计与文档整理。Web 后端 13 个测试类、116 个用例全部通过，核心包 Controller、Service 和 Util 覆盖率较高；Web 前端 3 个测试文件、46 个用例全部通过，utils、api、router 的语句与分支覆盖率达到 100%；鸿蒙端已完成 10 个 Hypium 测试文件、约 145 个用例的编写与注册，提交前需补充真实运行结果。
