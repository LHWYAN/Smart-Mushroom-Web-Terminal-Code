## 1 引言

### 1.1 目的

本文档是"智慧菌类养殖环境监控与管理系统"的详细设计报告，旨在为实训验收、后续维护与功能扩展提供完整、可执行的设计依据。

系统面向蘑菇大棚等菌类养殖场景，通过 **WS63 嵌入式终端** 采集温湿度、光照、烟雾等环境数据，经 **华为云 IoTDA** 平台完成设备接入与消息转发；**Spring Boot 后端** 消费 AMQP 消息并持久化至 **人大金仓 KingbaseES** 数据库；**Vue 3 Web 端** 提供监控大屏、数据 CRUD、用户认证与 AI 种植问答；**HarmonyOS 移动端** 通过 IoTDA 应用侧 API 实现设备影子查询与远程控制，并扩展告警、生长日记、环境配方等场景化功能。

文档重点描述：

- 三端协同的系统总体架构与数据/控制闭环；
- 功能模块划分及模块间交互关系；
- 金仓数据库四表设计与 ER 关系；
- 不少于四个核心模块的类设计、流程设计与接口说明。

## 2 系统架构设计

### 2.1 总体设计思想

系统采用 **"感知层 — 网络层 — 平台层 — 应用层 — 数据层"** 五层架构，以华为云 IoTDA 为中枢，实现菌类养殖环境数据从采集、上报、存储到多端展示的完整闭环。各层职责清晰、接口标准化，支持 Web 端与鸿蒙端并行接入，嵌入式端仅负责感知与执行。

【**此处插入图片：图 2-1 系统总体架构图**  
内容要求：从左到右或从上到下展示五层——  
① 感知层：WS63 + DHT11/MQ-2/LDR/OLED/LED/蜂鸣器；  
② 网络层：Wi-Fi + MQTT；  
③ 平台层：华为云 IoTDA（设备影子、规则引擎、AMQP 队列）；  
④ 应用层：Spring Boot 后端、Vue 3 Web、HarmonyOS App；  
⑤ 数据层：KingbaseES（4 表）+ SQLite（开发备用）。  
需标注双路径：端侧 MQTT 上报、后端 AMQP 消费、鸿蒙应用侧 API 直连。】

### 2.2 分层架构说明

#### （1）感知层

**硬件平台**：WS63 开发板（原智慧房间 IoT 项目演进为蘑菇大棚场景）。

**外设清单**：

| 外设 | 功能 | 驱动模块 |
|------|------|----------|
| DHT11 | 温度、湿度采集 | `dht11/dht11.c` |
| LDR + ADC | 光照强度采集 | `adc/ldr.c` |
| MQ-2 | 烟雾浓度（百分比） | `smoke_sensor/mq2.c` |
| SSD1306 OLED | 本地实时显示 | `oled/oled.c` |
| LED | 补光灯模拟（SetLamp） | `led/led.c` |
| 有源蜂鸣器 | 告警输出（SetBuzzer） | `buzzer/buzzer.c` |

**运行模型**：FreeRTOS 风格多任务——`appmain_start` 任务负责 Wi-Fi 连接与传感器初始化；`environment_task` 任务以 100ms 周期循环采集并刷新 OLED。CO2 字段在端侧固定初始化为 `"400"`（展示字段，非真实 CO2 传感器采集）。

#### （2）网络层

端侧通过 Wi-Fi 接入局域网，使用 **MQTT over TLS** 连接华为云 IoTDA Broker。属性上报周期与 MQTT 发布任务绑定；命令订阅 Topic 接收云端下发的 SetLamp/SetCond/SetVent/SetBuzzer 四类控制指令。

#### （3）平台层

华为云 IoTDA 承担：

- 设备注册与物模型管理（服务 ID：`smartRoom`）；
- 属性上报接收与设备影子更新；
- 命令下发与响应回传；
- 规则引擎将属性变化转发至 **AMQP 队列** `roomone-data`，供 Web 后端消费。

#### （4）应用层

| 终端 | 技术栈 | 主要职责 |
|------|--------|----------|
| Web 后端 | Java 17 + Spring Boot 3.2.5 | AMQP 消费、REST API、WebSocket 广播、用户认证、AI 代理、数据 CRUD |
| Web 前端 | Vue 3 + Vite 6 + Element Plus + ECharts | 登录注册、监控仪表盘、设备/数据/命令管理、AI 问答 |
| HarmonyOS | ArkTS + 一次开发多端部署 | IoT 设备影子查询、远程控制、告警/生长/配方/地图等场景页 |
| Python AI 服务 | PyTorch + Paho MQTT | 摄像头截图、MushroomCNN 病害识别、plantAI 属性上报 |

**重要架构说明**：

- Web 后端**不直连硬件**，仅通过 AMQP 间接获取 IoTDA 转发的 `smartRoom` 环境数据；
- HarmonyOS **不经过本地 Spring Boot**，直接调用 IoTDA **应用侧 REST API**（IAM Token 鉴权）；
- **CNN 病害识别**为独立 Python 进程，设备侧 MQTT 订阅 `recognizeOnce` 命令，模型权重见 `D:\LHWYAN\download\environment\Python-AI\mushroom_cnn.pth`；
- **Dify 种植问答**由 Web 端 `/ai` 页面经 Spring Boot 代理调用，与 CNN 识别并列存在。

#### （5）数据层

- **生产/验收**：KingbaseES V009R001，数据库名 `smart_room`，共 **4 张业务表**；
- **开发调试**：SQLite 文件 `./sensor_data.db`，业务三表结构一致；
- **双数据源**：MyBatis-Plus 操作 `device_info`/`sensor_data`/`device_commands`；`AuthService` 通过原生 kingbase8 驱动 + JdbcTemplate 独立操作 `users` 表。

### 2.3 数据流与控制流

#### 2.3.1 数据采集流

```
WS63 传感器采集
    → environment_task 更新全局变量
    → mqtt.c 组装 JSON（smartRoom 服务属性）
    → MQTT Publish → IoTDA 设备影子
    → IoTDA 规则引擎 → AMQP 队列
    → DataConsumerManager 解析消息
    → LatestDataHolder 内存快照 + SensorDataService 入库
    → RealtimeWebSocketHandler 广播 sensor_update
    → Vue Dashboard 实时刷新
```

【**此处插入图片：图 2-2 数据采集时序图**  
参与者：WS63、IoTDA、AMQP、DataConsumerManager、Database、WebSocket、Vue 前端。】

#### 2.3.2 设备控制流（双路径）

**路径 A — HarmonyOS 直连 IoTDA**：

```
用户点击控制按钮
    → DashboardViewModel.toggleControl()
    → IotService.sendCommand(SetLamp/SetCond/SetVent)
    → IAM 鉴权 + IoTDA 应用侧 API
    → IoTDA 命令下发 → MQTT → WS63 handle_cloud_command()
    → 执行器动作 + 命令响应 JSON 回传
```

**路径 B — Web 端命令日志（当前实现）**：

```
用户在命令管理页录入命令
    → CommandController → device_commands 表 CRUD
    → 状态流转 pending → success/fail
（注：Web 端当前以命令日志管理为主，未直接调用 IoTDA 下发接口）
```

标注 HarmonyOS 实时控制路径 vs Web 命令日志管理路径。】

### 5.1 模块一：嵌入式 MQTT 通信模块

#### 5.1.1 模块职责

负责 WS63 与华为云 IoTDA 之间的双向 MQTT 通信：周期性上报 `smartRoom` 服务属性；订阅命令 Topic 并解析执行；回传命令执行结果。

#### 5.1.2 类/文件设计

本模块为 C 语言实现，核心函数如下：

| 函数/变量 | 类型 | 职责 |
|-----------|------|------|
| `environment_task` | 任务入口 | 100ms 周期采集，更新全局状态 |
| `Setup_JSON_Tree_JX` | 静态函数 | 组装上报 JSON |
| `mqtt_publish` | 公开函数 | 发布属性消息 |
| `messageArrived` | MQTT 回调 | 接收命令消息 |
| `handle_cloud_command` | 静态函数 | 解析并执行四类命令 |
| `LampSt/CondiSt/VentSt/BuzzerSt` | 全局变量 | 执行器状态字符串 |
| `SmokeSt/CO2St` | 全局变量 | 烟雾与 CO2 展示值 |

#### 5.1.3 属性上报 JSON 格式

物模型服务 ID 为 `smartRoom`，属性字段与后端/鸿蒙三端对齐：

```json

      "CondST": "OFF",
      "VentST": "OFF",
      "BuzzerST": "OFF",
      "Smoke": "3",
      "CO2": "400"
    },
    "event_time": ""
  }]
}
```

#### 5.1.4 命令处理设计

| command_name | paras 键 | 合法值 | 端侧动作 |
|--------------|----------|--------|----------|
| SetLamp | LampStatus | ON/OFF | 控制 LED，`lampState` 更新 |
| SetCond | CondStatus | ON/OFF | 更新空调状态变量 `condiState` |
| SetVent | VentStatus | ON/OFF | 更新排风状态变量 `ventState` |
| SetBuzzer | BuzzerStatus | ON/OFF | 蜂鸣器开关 |

命令响应格式：

```json
{"result_code": 0, "response_name": "SetLamp", "paras": {"Result": "success"}}
```

#### 5.1.6 异常处理

- Topic/Payload 长度越界：截断保护，打印告警日志；

#### 5.2.1 模块职责

### 5.3 模块三：Web 用户认证模块

#### 5.3.1 模块职责

#### 5.3.3 双数据源架构

```
                    ┌─────────────────────────┐

                    └───────────┬─────────────┘
```

#### 5.3.4 认证流程

**注册流程**：

1. 前端 POST `/api/v1/auth/register`，Body：`{username, password, nickname?}`；
2. `AuthService.register` 检查用户名唯一性；
3. BCrypt 编码密码，INSERT 至 `users` 表，role=`user`，status=`active`；
4. 返回用户信息（不含密码）。

**登录流程**：

1. POST `/api/v1/auth/login`；
2. 校验用户名存在、status=active、密码匹配；
3. `JwtUtil.generateToken` 生成 UUID Token，有效期 24h，同用户旧 Token 被移除；
4. 前端 `setAuth(token, user)` 写入 localStorage，后续请求携带 `Authorization: Bearer {token}`。

**路由守卫**：

- 访问 `/dashboard` 等需鉴权路由时，若 `!isLoggedIn()` 则跳转 `/login`；
- 已登录用户访问 `/login` 或 `/register` 则重定向 `/dashboard`。

### 5.5 模块五：CNN 蘑菇病害识别模块

#### 5.5.1 模块职责

#### 5.5.2 模型与文件

| 项目 | 说明 |
|------|------|
| 模型权重 | `D:\LHWYAN\download\environment\Python-AI\mushroom_cnn.pth` |

| 训练相关 | 同目录下 `mushroom_train.ipynb`、`mushroom_test.ipynb` |

```

  → Conv2d(64)+ReLU+MaxPool+Dropout
  → Conv2d(128)+ReLU+MaxPool+Dropout
  → Flatten → Linear(18432→512)+ReLU+Dropout → Linear(512→3)
  → Softmax → 3 类概率
```

| 属性/命令 | 名称 | 说明 |

| 属性 | confidence | 置信度（0~1） |
| 属性 | captureTime | 截图时间 |
| 属性 | cameraStatus | 摄像头状态（在线/异常） |
| 属性 | aiStatus | AI 状态（完成/失败） |

#### 5.5.5 处理流程

```
IoTDA 控制台下发 recognizeOnce
  → Python on_message 回调
  → capture_one_frame() 从 CAMERA_URL 截图至 captures/
  → ai_predict() 加载 mushroom_cnn.pth 推理
```

## 7 数据库设计

### 7.1 设计概述

本系统在金仓 KingbaseES 上建立 **4 张业务表**，满足实训"3 表以上"要求：

| 表名 | 用途 | 访问方式 |
|------|------|----------|
| `device_info` | IoT 设备注册信息 | MyBatis-Plus |
| `sensor_data` | 传感器历史数据 | MyBatis-Plus |
| `device_commands` | 设备控制命令日志 | MyBatis-Plus |
| `users` | 用户账户 | JdbcTemplate + kingbase8 原生驱动 |

开发环境默认数据源为 SQLite，表结构通过 `schema-sqlite.sql` / `init-kingbasees.sql` 初始化；验收时将 `app.db-type` 切换为 `kingbasees`，四表全部落地金仓。

### 7.3 表结构详细设计

#### 7.3.1 device_info（设备信息表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | SERIAL / INTEGER | PK, 自增 | 主键 |
| device_id | VARCHAR(100) | NOT NULL, UNIQUE | 设备逻辑 ID，如 `roomone` |
| device_name | VARCHAR(200) | NOT NULL, DEFAULT '' | 设备名称 |
| device_type | VARCHAR(50) | DEFAULT 'sensor' | 设备类型 |
| location | VARCHAR(200) | DEFAULT '' | 安装位置 |
| status | VARCHAR(50) | DEFAULT 'offline' | online/offline |
| remarks | TEXT | DEFAULT '' | 备注 |
| create_time | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 更新时间 |

**默认数据**：`roomone` — "1号蘑菇大棚"，位于"沈阳网联实验室"。

#### 7.3.2 sensor_data（传感器数据表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | SERIAL | PK | 主键 |
| device_id | VARCHAR(100) | FK → device_info | 所属设备 |
| Temp | VARCHAR(50) | 温度 | 如 "26.5" |
| Humi | VARCHAR(50) | 湿度 | 如 "68.0" |
| Lumi | VARCHAR(50) | 光照 | ADC 原始值或 lux |
| LampST | VARCHAR(20) | 补光灯状态 | ON/OFF |
| CondST | VARCHAR(20) | 空调状态 | ON/OFF |
| VentST | VARCHAR(20) | 排风状态 | ON/OFF |
| BuzzerST | VARCHAR(20) | 蜂鸣器状态 | ON/OFF |
| Smoke | VARCHAR(50) | 烟雾浓度 | 百分比字符串 |
| CO2 | VARCHAR(50) | CO2 浓度 | 展示字段 |
| create_time | TIMESTAMP | 采集时间 | 索引字段 |

**索引**：`idx_sensor_data_device (device_id, create_time DESC)`

> 字段命名与 IoT 物模型保持一致（PascalCase），便于 AMQP 消息直写。

#### 7.3.3 device_commands（命令记录表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | SERIAL | PK | 主键 |
| device_id | VARCHAR(100) | FK | 目标设备 |
| command | VARCHAR(100) | NOT NULL | SetLamp/SetCond/SetVent/SetBuzzer |
| param_key | VARCHAR(100) | DEFAULT '' | 参数键名 |
| param_value | VARCHAR(100) | DEFAULT '' | ON/OFF |
| status | VARCHAR(50) | DEFAULT 'pending' | pending/success/fail |
| create_time | TIMESTAMP | 创建时间 | |
| update_time | TIMESTAMP | 更新时间 | |

**索引**：`idx_commands_device (device_id, create_time DESC)`

#### 7.3.4 users（用户信息表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | SERIAL | PK | 主键 |
| username | VARCHAR(100) | NOT NULL, UNIQUE | 登录名 |
| password | VARCHAR(255) | NOT NULL | BCrypt 哈希 |
| nickname | VARCHAR(100) | DEFAULT '' | 显示昵称 |
| role | VARCHAR(20) | DEFAULT 'user' | user/admin |
| status | VARCHAR(20) | DEFAULT 'active' | active/disabled |
| create_time | TIMESTAMP | 注册时间 | |
| update_time | TIMESTAMP | 更新时间 | |

**初始化**：系统启动时 `AuthService.initTable()` 自动建表，并创建默认管理员 `admin / admin123`。

### 7.4 数据完整性设计

| 规则 | 实现方式 |
|------|----------|
| 设备 ID 唯一 | `device_info.device_id` UNIQUE 约束 |
| 外键引用 | sensor_data / device_commands 引用 device_info(device_id) |
| 级联删除 | 业务层 `DeviceService.delete()` 手写事务级联，非 DB CASCADE |
| 用户名唯一 | users.username UNIQUE + 注册前查询 |
| 密码安全 | BCrypt 加盐，接口返回不含 password 字段 |

## 10 非功能性设计

### 10.1 安全性

| 措施 | 说明 |
|------|------|
| 密码哈希 | BCrypt 加盐存储 |
| 接口鉴权 | Bearer Token，业务页需登录 |
| 凭据隔离 | IoT/Dify/金仓密码放配置文件，HarmonyOS IotConfig gitignore |
| CORS | `CorsConfig` 允许前端跨域 |

### 10.2 可靠性

| 措施 | 说明 |
|------|------|
| AMQP 重连 | 指数退避，最大 60 秒 |
| IoT 降级 | HarmonyOS IoT 失败时使用 Mock 数据 |
| WebSocket + 轮询 | 双通道保证监控数据可达 |
| 事务 | 设备级联删除 `@Transactional` |

### 10.3 可测试性

| 端 | 框架 | 覆盖模块 |
|----|------|----------|
| 后端 | JUnit 5 + JaCoCo | Controller/Service/Util |
| 前端 | Vitest | api/router/utils |
| HarmonyOS | Hypium | ViewModel/ChartUtils/Theme |

### 10.4 性能考虑

- AMQP 入库间隔 30 秒，避免高频写库；
- 内存快照 `LatestDataHolder` 提供 O(1) 最新值查询；
- WebSocket 广播仅推送变化数据，减少前端轮询压力。
