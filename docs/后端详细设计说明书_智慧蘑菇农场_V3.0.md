## 1 引言

### 1.1 目的

本文档是"智慧蘑菇农场物联网后端服务"的详细设计说明书。系统以 WS63/Hi3863 端侧设备采集的温湿度、光照、烟雾等环境数据为基础，通过华为云 IoTDA 完成设备接入、属性上报和消息转发；本地后端服务采用 Java 17 + Spring Boot 3 架构，通过 AMQP 1.0 消费设备上报数据，经过业务层处理后写入数据库，并向 Vue 3 前端提供实时监控、历史数据查询、设备管理、传感器数据管理、命令管理、用户认证与 AI 种植问答等功能。

文档重点说明后端侧的设计实现，并**着重描写国产数据库（人大金仓 KingbaseES）的应用**，包括原生驱动接入、双数据源架构、四张业务表的建表与读写、用户认证数据直接落库金仓等。其核心目标是证明本项目满足学校实训要求：基于国产数据库（金仓数据库）开发项目；能够在 KingbaseES 上建立 3 个以上数据表（实际建立 4 张）；在 Web 端至少完成一个与项目相关的业务功能，并包含完整增删改查。

### 1.2 读者对象

本文档面向实训指导教师、后端开发人员、前端联调人员、测试人员和后续维护人员。指导教师可据此审查国产数据库使用情况、后端业务功能完整性和文档规范性；开发人员可依据模块说明维护代码；测试人员可依据接口设计和验收用例进行功能验证；前端联调人员可依据 REST API、WebSocket 和 AI 接口说明完成页面对接。

## 2 项目背景与需求对应

### 2.1 端侧系统与后端关系

本项目的前身是智慧房间 IoT 项目，原有 WS63/Hi3863 端侧设备作为蘑菇农场环境采集终端继续使用；新版后端和前端已独立重构，业务场景从通用房间监控聚焦为蘑菇种植环境监控。端侧设备主要外设包括 DHT11 温湿度传感器、光敏电阻模块、MQ-2 烟雾传感器、OLED 显示屏、LED 和有源蜂鸣器，可采集温湿度、光照、烟雾等环境数据，并按照平台模型上报 CO2 等展示字段。数据通过 MQTT 协议上报至华为云 IoTDA 平台，设备同时监听云端下发的 SetLamp、SetCond、SetVent、SetBuzzer 等控制命令。

新版后端不直接连接硬件，而是从华为云 IoTDA 转发出的 AMQP 队列消费设备上报数据。新版项目采用 Java Spring Boot 3 重写，保持 AMQP 消费逻辑和 RESTful 接口规范，同时将前端升级为 Vue 3 + Element Plus 独立项目，并增加用户认证登录与 AI 种植问答功能。其中用户认证模块的数据直接写入金仓数据库 `users` 表，进一步强化国产数据库的应用深度。

## 3 系统总体设计

### 3.1 总体架构

系统采用"端侧设备层—云平台接入层—后端服务层—数据持久层—前端展示层"的分层架构。端侧设备负责采集蘑菇种植环境数据并执行控制命令；华为云 IoTDA 负责设备接入、规则引擎和 AMQP 队列转发；Spring Boot 后端负责消息消费、业务接口、用户认证、AI 接入和 WebSocket 广播；KingbaseES（金仓数据库）作为验收和生产主数据库负责数据持久化，承载四张业务表；Vue 3 前端负责业务展示和用户交互，登录前不可访问业务页面。

后端数据持久层采用**双数据源架构**：默认数据源（由 Spring Boot 自动配置，开发环境为 SQLite）供 MyBatis-Plus 操作 `device_info`、`sensor_data`、`device_commands` 三张业务表；`KingbaseConfig` 单独定义 `kingbaseDataSource` Bean，使用原生 `kingbase8` 驱动与 HikariCP 连接池，供 `AuthService` 通过 `JdbcTemplate` 操作 `users` 表。验收时将默认数据源也切换至金仓，使四张表全部落地 KingbaseES。

### 3.2 数据流与控制流

采集流从 WS63 设备开始：设备通过 MQTT 将属性上报至 IoTDA，IoTDA 规则引擎将设备属性转发到 AMQP 队列；后端 `DataConsumerManager` 通过 Apache Qpid JMS 消费 AMQP 消息，解析后更新 `LatestDataHolder` 内存快照，并按采样间隔调用 `SensorDataService` 写入数据库，同时通过 `RealtimeWebSocketHandler` 广播 `sensor_update` 事件给已连接前端。

控制流从 Vue 前端开始：用户**必须先登录**获取 Token，前端路由守卫拦截未登录请求并跳转至 `/login`；登录成功后用户通过设备管理、数据管理和命令管理页面发起操作，前端请求携带 `Authorization: Bearer {token}` 请求头，Controller 接收 REST 请求后调用 Service 和 Mapper 完成数据库操作。命令管理当前作为命令日志 CRUD 和状态流转功能。AI 问答流由前端 `/ai` 页面发起，后端调用 Dify API 获取回答，经 `AiAnswerSanitizer` 清洗思考链后返回。

## 4 功能设计

系统围绕 `device_info`、`sensor_data`、`device_commands`、`users` 四张业务表组织基础业务，并在新版中增加用户认证登录、实时 WebSocket 推送和 AI 种植问答能力。设备管理、传感器数据管理、命令管理、用户注册登录共同满足数据库增删改查要求；实时监控和 AI 问答体现智慧蘑菇农场场景价值。

### 4.1 用户认证与权限功能

用户认证对应 `users` 表，是 V3.0 新增的核心功能，也是国产数据库应用的重点。该功能包含注册、登录、获取当前用户信息、退出登录四个子功能，所有数据直接写入金仓数据库 `users` 表。

注册操作要求 `username` 唯一，密码经 BCrypt 加盐哈希后存储；登录操作校验用户名、密码和账户状态，验证通过后由 `JwtUtil` 生成 UUID Token 并在内存中维护 `token→username` 与 `username→token` 双向映射，同一用户重复登录会踢掉旧 Token（单设备登录）。获取当前用户信息接口要求请求头携带 `Authorization: Bearer {token}`，后端校验 Token 有效性与过期时间。退出登录接口移除服务端 Token 记录。

系统启动时由 `AuthService.initTable()` 通过 `CREATE TABLE IF NOT EXISTS users` 在金仓中自动建表，并初始化默认管理员账户 `admin / admin123`。前端通过 Vue Router 导航守卫 `beforeEach` 拦截未登录用户，未登录访问业务页面时跳转至 `/login`；登录态通过 `localStorage` 持久化 Token 和用户信息，并在 Axios 请求拦截器中统一注入 `Authorization` 请求头。

### 4.2 设备管理功能

设备管理对应 `device_info` 表，用于维护 IoT 设备注册信息。新增操作要求 `device_id` 唯一；查询操作支持设备列表和单设备查询；修改操作允许更新设备名称、类型、安装位置、在线状态和备注；删除设备时，由 `DeviceService.delete()` 在业务层手写级联删除该设备关联的 `sensor_data` 和 `device_commands` 记录，使用 `@Transactional` 保证事务一致性，而不是依赖数据库 `ON DELETE CASCADE`。

### 4.5 实时监控功能

实时监控页面为 Vue 3 单页组件，路由为 `/dashboard`。页面包含大棚头部、运行状态指示灯、告警栏、概览统计、实时传感器卡片、趋势图表和最新采集记录。实时卡片展示温度、湿度、CO2、光照、烟雾五项指标；告警栏可根据烟雾浓度阈值显示异常提示；趋势图基于 ECharts 绘制 24 小时温度和湿度双折线图；最新记录使用 Element Plus 表格展示。

## 5 数据库设计

> 本章是国产数据库应用的核心章节。本系统在金仓数据库 KingbaseES V009R001 上建立 4 张业务表，覆盖设备、传感器数据、命令记录和用户账户四类业务实体，所有业务功能（含增删改查）均落地金仓。

### 5.1 数据库连接设计

本系统的数据库设计以国产金仓数据库 KingbaseES V009R001 为验收主场景。后端采用**双数据源架构**连接金仓：

**数据源一：业务数据源（MyBatis-Plus）**
由 Spring Boot 自动配置，开发环境默认为 SQLite（`jdbc:sqlite:./sensor_data.db`），验收时通过 `app.db-type=kingbasees` 切换。该数据源供 MyBatis-Plus 的 `DeviceInfoMapper`、`SensorDataMapper`、`DeviceCommandMapper` 操作 `device_info`、`sensor_data`、`device_commands` 三张表。连接参数由 `spring.datasource` 配置。

**数据源二：金仓认证数据源（JdbcTemplate）**
由 `KingbaseConfig` 配置类手动定义，使用**人大金仓官方原生驱动 `com.kingbase8.Driver`**，连接串格式为 `jdbc:kingbase8://{host}:{port}/{database}`，通过 HikariCP 连接池管理（最大连接数 5，最小空闲 1，连接测试 SQL 为 `SELECT 1`）。该数据源供 `AuthService` 通过 `JdbcTemplate` 操作 `users` 表，**无论 `app.db-type` 如何配置，认证数据始终直连金仓数据库**。

金仓连接参数统一配置在 `application.yml` 的 `app.kingbase.*` 配置项中，主要包括 `host`（默认 127.0.0.1）、`port`（默认 54321）、`database`（默认 smart_room）、`username`（默认 system）、`password`。

`KingbaseConfig` 关键代码如下：

```java
@Configuration
public class KingbaseConfig {

    @Bean
    public DataSource kingbaseDataSource(AppProperties appProperties) {
        var k = appProperties.getKingbase();
        HikariDataSource ds = new HikariDataSource();
        ds.setDriverClassName("com.kingbase8.Driver");
        ds.setJdbcUrl("jdbc:kingbase8://" + k.getHost() + ":" + k.getPort() + "/" + k.getDatabase());
        ds.setUsername(k.getUsername());
        ds.setPassword(k.getPassword());
        ds.setMaximumPoolSize(5);
        ds.setMinimumIdle(1);
        ds.setConnectionTestQuery("SELECT 1");
        return ds;
    }

    @Bean
    public JdbcTemplate kingbaseJdbcTemplate(DataSource kingbaseDataSource) {
        return new JdbcTemplate(kingbaseDataSource);
    }
}
```

验收运行时将 `app.db-type` 设置为 `kingbasees`，启动服务后应通过启动日志和 `/api/v1/health` 接口确认当前数据库类型为 `kingbasees`，并在 KStudio 或 ksql 中验证 `device_info`、`sensor_data`、`device_commands`、`users` 四张业务表已经创建并可读写。

```sql
CREATE TABLE IF NOT EXISTS users (

    update_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP

```

| 字段名 | KingbaseES 类型 | 说明 |

## 7 模块设计

### 7.1 模块交互模型

新版后端运行时由 Spring Boot 容器统一管理。`SmartMushroomApplication` 启动后，配置类完成跨域、WebSocket、应用属性、金仓数据源等初始化；`AuthService` 通过 `@PostConstruct initTable()` 自动在金仓中创建 `users` 表并初始化管理员；`DataConsumerManager` 通过 `@PostConstruct start()` 自动启动数据源线程；HTTP 请求由 Controller → Service → Mapper 三层处理；AMQP 或模拟数据进入后调用 `SensorDataService` 入库、更新 `LatestDataHolder`，并通过 `RealtimeWebSocketHandler` 广播给前端。

### 7.3 Service 与 Mapper 层

Service 层负责业务校验、状态处理、统计聚合、手写级联删除和外部接口调用。Mapper 层使用 MyBatis-Plus，三个实体 `DeviceInfo`、`SensorData`、`DeviceCommand` 分别映射到 `device_info`、`sensor_data`、`device_commands` 表，对应 Mapper 继承 `BaseMapper<T>` 后可获得 `insert`、`deleteById`、`selectList`、`updateById` 等通用 CRUD 方法。

| 旧版 database.py 函数 | 新版 Mapper/Service 对应实现 |
|------------------------|------------------------------|
| get_connection() | 由 Spring Boot DataSource 自动管理。 |
| init_database() | KingbaseES 由 schema-kingbasees.sql 与 AuthService.initTable() 初始化。 |
| create_device() | DeviceInfoMapper.insert(device) 或 DeviceService.create()。 |
| get_all_devices() | DeviceInfoMapper.selectList(wrapper)。 |
| update_device() | DeviceService.update() 配合 MyBatis-Plus Wrapper。 |
| delete_device() | DeviceService.delete()，业务层手写级联删除关联数据。 |

### 7.4 DataConsumerManager 数据消费模块

`DataConsumerManager` 是新版数据消费核心模块，负责 AMQP 与模拟数据源的生命周期管理。默认模拟模式下使用 `ScheduledExecutorService` 定时生成传感器数据；AMQP 模式下使用 `JmsConnectionFactory` 连接华为云 IoTDA AMQP 1.0 队列，轮询接收 `TextMessage`，解析 `device_id`、`notify_data.body` 或 `services[i].properties` 中的属性字段，更新最新快照、广播 WebSocket，并按 `app.amqp-db-interval` 采样间隔入库。

| 方法/职责 | 说明 |
|-----------|------|
| start() / stop() | 通过 @PostConstruct / @PreDestroy 完成消费线程启停。 |
| startSimulateConsumer() | 启动模拟数据定时任务。 |
| simulateOnce() | 生成一组随机传感器数据，用于开发演示。 |
| startAmqpConsumer() / runAmqpLoop() | 启动 AMQP 消费循环，连接、接收、解析、入库和重连。 |
| parseAmqpMessage(body, deviceId) | 解析 IoTDA JSON 消息，提取设备属性字段。 |
| persistData(data, markOnline) | 完成入库、在线状态更新、最新快照更新和 WebSocket 广播。 |

### 7.5 WebSocket 实时推送模块

`WebSocketConfig` 注册 `/ws/realtime` 端点，`RealtimeWebSocketHandler` 维护已连接客户端集合。`DataConsumerManager` 在接收到新数据后调用 `broadcast()` 广播 `sensor_update` 事件，使前端仪表盘可在轮询之外获得更及时的数据刷新。

### 7.6 用户认证模块

用户认证模块是 V3.0 新增模块，由 `AuthController`、`AuthService`、`JwtUtil`、`User` 实体与 `KingbaseConfig` 数据源协同实现，**全部数据操作直接落在金仓数据库 `users` 表上**。

`AuthService` 注入 `JdbcTemplate kingbaseJdbcTemplate`（由 `KingbaseConfig` 提供的金仓专用 JdbcTemplate），通过原生 SQL 实现 `users` 表的增删改查：注册时执行 `INSERT INTO users ...` 并用 BCrypt 加密密码；登录时执行 `SELECT * FROM users WHERE username = ?` 查询用户，用 `BCryptPasswordEncoder.matches()` 校验密码；查询用户通过 `RowMapper<User>` 将结果集映射为 `User` 对象。

`JwtUtil` 采用 UUID + 内存 `ConcurrentHashMap` 方案管理 Token，维护 `token→TokenEntry(username, expireAt)` 与 `username→token` 双向映射，Token 有效期 24 小时，同一用户登录会移除旧 Token 实现单设备登录。

| 方法/职责 | 说明 |
|-----------|------|
| initTable() | @PostConstruct 启动时在金仓创建 users 表并初始化 admin 账户。 |
| register(username, password, nickname) | 注册新用户，BCrypt 加密密码后写入金仓 users 表。 |
| login(username, password) | 登录校验，验证用户名、密码与账户状态。 |
| getUserByUsername(username) | 按用户名查询用户（含密码 hash）。 |
| generateToken(user) | 调用 JwtUtil 生成 Token。 |
| getCurrentUser(token) | 校验 Token 并返回当前用户（不含密码）。 |
| logout(token) | 移除服务端 Token 记录。 |

### 7.7 AI 服务模块

`AiService` 通过 `RestTemplate` 调用 Dify API 的聊天消息接口，使用 Bearer Token 鉴权，阻塞等待大模型返回回答。返回的 `answer` 字段经 `AiAnswerSanitizer.sanitize()` 清洗，去除 DeepSeek-R1 等推理模型的 `<think>`、`<thinking>`、`<redacted_thinking>` 思考链标签（含成对标签、孤立开标签、孤立闭标签三种情况），并折叠多余空行。`AiController` 暴露 `POST /api/v1/ai/chat` 接口，接收包含 `question` 的 JSON 请求，返回 `answer` 字段。异常时记录日志并返回错误提示，前端应展示"AI 助手暂时不可用"等友好信息。

`AiAnswerSanitizer` 关键清洗逻辑：

```java
// 成对标签：<think>...</think>
text = THINKING_BLOCKS.matcher(text).replaceAll("");
// 孤立开标签：<think>... （无闭合）
text = ORPHAN_OPEN_TAG.matcher(text).replaceAll("");
// 孤立闭标签：</think>
text = ORPHAN_CLOSE_TAG.matcher(text).replaceAll("");
// 折叠 3 行及以上空行为 2 行
text = EXCESS_BLANK_LINES.matcher(text).replaceAll("\n\n");
```

## 8 华为云 IoTDA 数据接入设计

### 8.1 设备属性上报字段映射

端侧属性上报 `service_id` 为 `smartRoom`，`properties` 中包含 Temp、Humi、Lumi、LampST、CondST、VentST、BuzzerST、Smoke、CO2 等字段。后端实体字段与上报字段保持一致，以降低解析和联调成本。

| 上报字段 | 数据库字段 | 含义 | 示例值 |
|----------|------------|------|--------|
| Temp | sensor_data.Temp | 温度 | 28.5 |
| Humi | sensor_data.Humi | 湿度 | 60.2 |
| Lumi | sensor_data.Lumi | 光照强度 | 75 |
| LampST | sensor_data.LampST | 照明灯状态 | ON |
| CondST | sensor_data.CondST | 空调状态 | OFF |
| VentST | sensor_data.VentST | 排风扇状态 | OFF |
| BuzzerST | sensor_data.BuzzerST | 蜂鸣器状态 | OFF |
| Smoke | sensor_data.Smoke | 烟雾浓度或告警值 | 0 |
| CO2 | sensor_data.CO2 | CO2 字段值 | 400 |

### 8.2 AMQP 消息处理流程

1. 使用 Apache Qpid JMS 连接 `amqps://{host}:{port}`，认证用户名格式为 `accessKey={key}|timestamp={ms}|instanceId={id}`。
2. `runAmqpLoop()` 中通过 `consumer.receive(1000)` 轮询获取消息。
3. 收到 `TextMessage` 后调用 `parseAmqpMessage()`，使用 Jackson 将 JSON 解析为 `JsonNode`。
4. 提取 `device_id`；如果设备 ID 包含下划线，可取最后一段作为本地 `device_id`，例如 `xxx_roomone` 映射为 `roomone`。
5. 优先从 `notify_data.body` 或 `services[i].properties` 中提取 Temp、Humi、Lumi、Smoke、CO2 和设备状态字段。
6. 更新 `LatestDataHolder` 内存快照，并通过 `RealtimeWebSocketHandler.broadcast()` 向所有 WebSocket 客户端推送。
7. 按 `app.amqp-db-interval` 默认 30 秒的采样间隔调用 `SensorDataService.insert()` 入库。
8. 解析异常或连接异常时记录日志并继续重试，避免单条异常消息影响消费线程。

## 9 接口设计

## 10 部署与运行设计

### 10.1 部署拓扑

开发与验收环境可采用单机部署：后端 Spring Boot 服务、前端 Vite 开发服务器和 KingbaseES 数据库均运行在同一台 Windows 或 Linux 电脑上。WS63 设备通过 WiFi 接入互联网并连接华为云 IoTDA；浏览器可在本机或局域网其他设备访问前端页面。AI 功能需要后端能够访问 Dify API 地址。

### 10.2 运行环境要求

| 环境项 | 要求 |
|--------|------|
| 操作系统 | Windows 10/11 或 Linux。 |
| Java | JDK 17 及以上。 |

| 浏览器 | Chrome、Edge 或其他现代浏览器，支持 WebSocket 和 ES6。 |

### 10.3 部署步骤

**后端部署**：进入 `backend` 目录，执行 `mvn spring-boot:run`；或执行 `mvn clean package` 后使用 `java -jar target/*.jar` 启动。服务默认运行在 `http://localhost:8080`。首次启动时 `AuthService` 会自动在金仓中创建 `users` 表并初始化 `admin/admin123` 账户。

**前端部署（开发模式）**：进入 `frontend` 目录，执行 `npm install` 安装依赖，再执行 `npm run dev` 启动 Vite 开发服务器，访问 Vite 控制台输出的地址（通常为 `http://localhost:5173`）。开发时 Vite 将 `/api` 代理到后端 8080 端口。

**前端部署（生产模式，可选）**：执行 `npm run build` 生成 `frontend/dist` 静态资源，可部署到独立 HTTP 服务器，也可由后端静态资源服务托管。

**KingbaseES 启用**：验收时修改 `application.yml`，将 `app.db-type` 设置为 `kingbasees`，配置 `app.kingbase.*` 参数，并在 KStudio 或 ksql 中执行 `schema-kingbasees.sql`（`users` 表由后端启动时自动创建）；重启后通过启动日志和 `/api/v1/health` 验证 `db_type=kingbasees`。

**访问验证**：浏览器访问 `/api/v1/health` 查看健康状态；访问 `/doc.html` 查看 Knife4j 接口文档；访问前端首页会自动跳转至登录页，使用 `admin/admin123` 登录后验证实时数据、CRUD 和 AI 页面。

## 12 安全、异常与运维设计

### 12.1 安全设计

| 风险点 | 设计措施 |
|--------|----------|
| 数据库密码泄露 | 数据库密码放入 application.yml 或环境变量，提交报告和截图时进行遮挡。 |
| 用户密码泄露 | 密码采用 BCrypt 加盐哈希存储，数据库不存明文；接口返回 User 对象时 password 字段置 null。 |
| Token 伪造 | Token 采用 UUID 服务端签发与校验，内存存储，无法伪造；有效期 24 小时，过期自动失效。 |
| 未授权访问 | 前端路由守卫拦截未登录请求；后端 profile 接口校验 Token。 |
| IoTDA 凭证泄露 | AMQP accessKey/accessCode 属敏感信息，文档中不展示完整值，截图需遮挡。 |
| Dify API Key 泄露 | app.dify.api-key 不应出现在公开文档或截图中，可使用环境变量覆盖。 |
| 非加密通信 | 端侧 MQTT 如使用 1883 端口仅适合演示，生产环境建议使用 TLS 端口并校验证书。 |
| 接口误操作 | 删除操作前端增加确认弹窗；后端可扩展登录鉴权后再对公网开放。 |
| SQL 注入 | MyBatis-Plus 参数绑定与 Wrapper 查询；AuthService 使用 JdbcTemplate 占位符 `?` 参数绑定，避免拼接用户输入。 |

### 12.2 异常处理设计

| 异常场景 | 处理方式 |
|----------|----------|
| 金仓连接失败 | 启动日志输出错误，AuthService 建表失败；检查数据库端口、账号、密码和 kingbase8 驱动是否就绪。 |
| 数据库连接失败 | 启动日志输出错误，健康检查返回异常；检查数据库端口、账号、密码和 db-type 配置。 |
| AMQP 连接失败 | DataConsumerManager 记录错误并按策略重连；前端仍可查询历史数据或使用模拟数据源演示。 |
| 消息格式不符合预期 | parseAmqpMessage 捕获解析异常，记录日志并跳过异常消息，不影响后续消费。 |
| 用户名已存在 | AuthService.register 返回 null，Controller 返回"注册失败，用户名可能已存在"。 |
| 登录密码错误 | BCrypt matches 校验失败，返回"用户名或密码错误"。 |
| Token 无效或过期 | JwtUtil.validateToken 返回 null，profile 接口返回"token无效或已过期"。 |
| 设备长时间不上报 | device_info.status 可保持 offline 或在超时逻辑中置为 offline；前端提示最近更新时间。 |
| WebSocket 断开 | 前端可自动重连或通过 HTTP 轮询保底刷新。 |
| Dify API 调用失败 | 后端记录日志，返回错误提示；前端显示 AI 助手暂时不可用。 |

### 12.3 运维日志设计

后端日志应至少包含服务启动、数据库初始化、金仓 users 表初始化、默认管理员创建、数据源启动、AMQP 连接成功/失败、模拟数据生成、消息解析异常、数据入库成功、WebSocket 客户端连接/断开、用户注册/登录、REST 接口错误和 Dify API 调用失败等信息。日志中不要打印完整 accessCode、数据库密码、Dify API Key 和用户密码哈希值。
